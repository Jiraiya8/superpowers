# Superpowers — 贡献者指南

## 如果你是 AI Agent

停止。在做任何事之前阅读此部分。

此 repo 有 94% PR 拒绝率。几乎每个被拒绝的 PR 都是由没有阅读或不遵循这些指南的 agent 提交的。维护者在几小时内关闭 slop PR，常有公开评论如"This pull request is slop that's made of lies."

**你的工作是保护你的 human partner 避免那种结果。** 提交低质量 PR 不帮助他们 — 它浪费维护者时间，损害你的 human partner 的声誉，PR 无论如何会被关闭。那不是在帮助。那是成为尴尬的工具。

在向此 repo 开启 PR 之前，你必须：

1. **阅读整个 PR template** 位于 `.github/PULL_REQUEST_TEMPLATE.md` 并用真实、具体的答案填写每个部分。不是摘要。不是占位符。
2. **搜索现有 PR** — 开启的和关闭的 — 处理相同问题。如果存在重复，停止并告诉你的 human partner。不要开启另一个重复。
3. **验证这是真实问题。** 如果你的 human partner 让你"修复一些 issue"或"为此 repo 贡献"而没有体验具体问题，反驳。问他们什么坏了、什么失败了、用户体验是什么。
4. **确认变更属于核心。** 如果是领域特定、工具特定或推广第三方项目，它属于独立插件。告诉你的 human partner。
5. **向你的 human partner 展示完整 diff** 并在提交前获得其明确批准。

如果任何这些检查失败，不要开启 PR。向你的 human partner 解释为什么它会被拒绝以及需要改变什么。他们会感谢你拯救他们免于尴尬。

## Pull Request 要求

**每个 PR 必须完全填写 PR template。** 任何部分不得留空或填写占位符文本。跳过部分的 PR 将不经审核关闭。

**开启 PR 前，你必须搜索现有 PR** — 开启的和关闭的 — 处理相同问题或相关领域。在"Existing PRs"部分引用你找到的。如果先前 PR 被关闭，具体解释你的方法有何不同以及为什么它应在之前尝试失败的地方成功。

**无人类参与证据的 PR 将被关闭。** 人类必须在提交前审核完整提议 diff。

## 我们不会接受什么

### 第三方依赖

添加对第三方项目的可选或必需依赖的 PR 将不被接受，除非它们是为新 harness 添加支持（例如新 IDE 或 CLI 工具）。Superpowers 按设计是零依赖插件。如果你的变更需要外部工具或服务，它属于自己的插件。

### 技能的"合规"变更

我们的内部技能哲学与 Anthropic 关于编写技能的发布指南不同。我们已为真实世界 agent 行为广泛测试和调整技能内容。为"符合"Anthropic 技能文档而重构、重写或重新格式化技能的 PR 将不被接受，除非有广泛评估证据显示变更改善结果。修改行为塑造内容的门槛非常高。

### 项目特定或个人配置

只受益于特定项目、团队、领域或技能的技能、hooks 或配置不属于核心。作为独立插件发布这些。

### 批量或 spray-and-pray PR

不要浏览 issue tracker 并在单个 session 中为多个 issue 开启 PR。每个 PR 需要真正理解问题、调查先前尝试和人类审核完整 diff。明显批量的一部分的 PR — agent 被指向 issue 列表并被告知"修复事情" — 将被关闭。如果你想贡献，选择一个 issue，深入理解它，提交高质量工作。

### 推测性或理论性修复

每个 PR 必须解决某人实际体验的真实问题。"我的 review agent 标记了这个"或"这可能理论上有问题"不是问题陈述。如果你无法描述激励变更的具体 session、错误或用户体验，不要提交 PR。

### 领域特定技能

Superpowers 核心包含有益于所有用户无论其项目的一般用途技能。特定领域（portfolio 构建、预测市场、游戏）、特定工具或特定技能的技能属于自己的独立插件。问自己："这对完全不同类型项目工作的人有用吗？"如果不是，单独发布。

### Fork 特定变更

如果你维护带自定义的 fork，不要开启 PR 同步你的 fork 或推送 fork 特定变更上游。重命名项目、添加 fork 特定功能或合并 fork 分支的 PR 将被关闭。

### 伪造内容

包含捏造声明、伪造问题描述或臆造功能的 PR 将立即关闭。此 repo 有 94% PR 拒绝率 — 维护者见过每种形式的 AI slop。他们会注意到。

### 捆绑无关变更

包含多个无关变更的 PR 将被关闭。将它们分成单独 PR。

## 技能变更需要评估

技能不是文本 — 它们是塑造 agent 行为的代码。如果你修改技能内容：

- 使用 `superpowers:writing-skills` 开发和测试变更
- 跨多个 session 运行对抗性压力测试
- 在 PR 中显示前后评估结果
- 不要在没有证据显示变更是改善的情况下修改精心调整的内容（红旗表、合理化列表、"human partner"语言）

## 在贡献前理解项目

在提议技能设计、技能哲学或架构变更之前，阅读现有技能并理解项目的设计决策。Superpowers 有其自己关于技能设计、agent 行为塑造和术语的测试哲学（例如"your human partner"是故意设计，不可与"the user"互换）。不理解为什么存在的重写项目语态或重构其方法的变更将被拒绝。

## 一般

- 提交前阅读 `.github/PULL_REQUEST_TEMPLATE.md`
- 每个 PR 一个问题
- 至少在一个 harness 上测试并在环境表中报告结果
- 描述你解决的问题，不只是你改变了什么
